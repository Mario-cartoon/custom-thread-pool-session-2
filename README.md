# Курсовая работа - Thread Pool

Курсовая работа представяет собой реализацию кастомного  CustomThreadPool с настраиваемым управлением очередями, логированием, параметрами и политикой отказа.


## Сборка и запуск

```bash
javac -d out src/com/poolcustom/*.java
java -cp out com.poolcustom.Main
```

## Что реализовано

- `execute(Runnable)` для выполнения обычных задач.
- `submit(Callable<T>)` для задач, которые возвращают результат через `Future<T>`.
- `shutdown()` для корректного завершения работы: новые задачи больше не принимаются, а уже добавленные выполняются до конца.
- `shutdownNow()` для немедленной остановки: задачи из очередей удаляются, а потоки прерываются.
- Подробное логирование всех основных событий в работе пула.

## Принцип работы

Пул хранит несколько независимых очередей, по одной на worker. Новая задача отправляется в очередь по алгоритму round robin. Если выбранная очередь переполнена, пул пытается создать новый worker с новой очередью, пока не достигнет `maxPoolSize`. Если и это невозможно, задача отклоняется.


## Анализ производительности

Эта реализация по производительности обычно уступает ThreadPoolExecutor из стандартной библиотеки Java. Причина в том, что стандартный пул давно оптимизирован, хорошо протестирован и лучше обрабатывает сложные и редкие ситуации.

Если сравнивать с промышленными решениями, например Tomcat или Jetty, разница будет ещё заметнее. Такие пулы рассчитаны на реальную высокую нагрузку: у них лучше настроены очереди задач, есть мониторинг состояния, гибкие политики отказа и более надёжная работа в составе больших серверных приложений.

## Наблюдения по параметрам
- `corePoolSize` лучше выбирать примерно под обычную постоянную нагрузку.
-  `maxPoolSize` можно увеличивать, если задачи часто ждут ввод-вывод или выполняются очень недолго. Но слишком большое число потоков может только замедлить работу.
-  Маленький `queueSize` быстрее показывает, что пул перегружен, потому что задачи начнут отклоняться раньше.
-  Большой `queueSize` позволяет реже отклонять задачи, но из-за этого они могут дольше стоять в очереди.
-  Маленький `keepAliveTime` быстрее завершает лишние временные потоки, но если нагрузка то появляется, то пропадает, потоки будут слишком часто создаваться заново.
-  `minSpareThreads` помогает держать несколько свободных потоков про запас, чтобы новые задачи начинали выполняться быстрее. Но слишком большое значение приводит к тому, что лишние потоки просто простаивают.

Обычно лучшая производительность получается не при максимальных значениях параметров, а при их разумном балансе.
- Для задач, которые сильно нагружают процессор, количество потоков обычно делают примерно равным числу ядер или немного больше.
- Для задач, которые часто ждут сеть, диск или базу данных, потоков можно сделать больше.
- Для коротких всплесков нагрузки обычно хорошо работает средний размер очереди и небольшой запас по maxPoolSize.
